# Multi-Author Tracked Changes — Design Record

**Status**: Decisions made. Ready for implementation.

---

## Problem

A document can have pending tracked changes from multiple independent sources
simultaneously: human reviewers, the document owner, and AI suggestions. All
three must coexist, remain independently accept/rejectable, and never silently
overwrite each other.

```
Original:   "The quick brown fox"

Bob:        del "quick "       tracked_delete:Bob
            ins "agile "       tracked_insert:Bob

Alice:      del "quick brown " tracked_delete:Alice   ← overlaps Bob on "quick "

AI:         del "brown"        tracked_delete:AI
            ins "golden"       tracked_insert:AI
```

This is not "collaborative editing" in the traditional sense. It is a
**Multi-Hypothesis Document State**: the document is a tree of concurrent
intentions, not a single agreed-upon string.

---

## Decision 1 — Data model: Split Ranges (Atomic Segments) ✅

**Chosen: Split the ranges. Each unique combination of authors = distinct text node segment.**

### Why not multi-author arrays inside a single mark

A `dataTracked: TrackedAttrs[]` array on a single inline mark seems cleaner in
JSON, but is a sync nightmare:

- If Bob deletes "The quick" and Alice deletes "quick brown", they overlap on
  "quick". To use a single mark you must merge them into one mark with two
  authors, then split them again if Bob retracts but Alice doesn't.
- This custom merge/split logic has to be re-run on every Yjs remote operation,
  fighting ProseMirror's native mapping.

### The split-ranges model

Break the text at every author-boundary so each segment has a clear,
single-or-double-author mark set.

```
"The quick brown fox"  + Bob deletes "The quick" + Alice deletes "quick brown":

Segment 1: "The "   → tracked_delete:Bob
Segment 2: "quick"  → tracked_delete:Bob  tracked_delete:Alice   ← conflict
Segment 3: " brown" → tracked_delete:Alice
Segment 4: " fox"   → (clean)
```

### Why this wins

1. **Yjs compatibility** — Yjs naturally tracks insertions between segments.
   No custom internal-array-offset logic needed on remote operations.
2. **Standard PM mapping** — if someone types elsewhere, ProseMirror's native
   step mapping shifts all small segments correctly.
3. **Accept logic is trivial** — "Accept Bob" = remove Bob's mark from every
   segment. If a segment has no marks left, it is clean.

### The ProseMirror unlock: `excludes: ""`

By default ProseMirror marks of the same type exclude each other (you cannot
have two `bold` marks on the same text node). Setting `excludes: ""` removes
that constraint, allowing two `tracked_delete` marks to coexist on the same
segment when two authors deleted exactly the same characters.

```typescript
// packages/plugins/src/track-changes/TrackChanges.ts

tracked_delete: {
  attrs: {
    dataTracked: { default: null },   // TrackedAttrs — single author per mark instance
  },
  inclusive: false,
  excludes: "",                        // ← allows multiple tracked_delete on same node
  parseDOM: [{ tag: "del[data-tracked]" }],
  toDOM() { return ["del", { "data-tracked": "delete" }, 0]; },
},

tracked_insert: {
  attrs: {
    dataTracked: { default: null },
  },
  inclusive: false,
  excludes: "",                        // ← same for insert
  parseDOM: [{ tag: "ins[data-tracked]" }],
  toDOM() { return ["ins", { "data-tracked": "insert" }, 0]; },
},
```

Each mark instance still carries a single `dataTracked` (one author, one id).
The split-range approach ensures you never need to merge two authors into a
single `dataTracked` object.

### `isConflict` flag on `TrackedAttrs`

When two authors' marks land on the same segment, both marks get
`isConflict: true`. This drives rendering (amber background) and the conflict
popover.

```typescript
// packages/plugins/src/track-changes/types.ts

export interface TrackedAttrs {
  id: string;
  authorID: string;
  operation: CHANGE_OPERATION;
  status: CHANGE_STATUS;
  createdAt: number;
  updatedAt: number;
  statusUpdateAt: number;
  reviewedByID: string | null;
  isConflict?: boolean;              // ← new: true when another author touched the same segment
  moveNodeId?: string;
}
```

## Mark bloat prevention — same-author dedup + post-accept merge

### The two bloat scenarios

**Scenario A — Duplicate mark on the same node**

With `excludes: ""`, ProseMirror no longer prevents two `tracked_delete` marks
of the same type from landing on the same text node. If the same author's mark
is applied twice (e.g. `appendTransaction` running more than once on the same
step, or a `splitRangeForNewMark` bug), you get:

```
"quick" → tracked_delete:Bob(id:"x")  tracked_delete:Bob(id:"x")  ← duplicate
```

ProseMirror's `Mark.eq` compares type + all attrs including `dataTracked.id`.
Exact duplicates (same id) are already deduplicated by ProseMirror's mark-set
normalisation — so this case is safe by default.

The dangerous variant is **same author, different ids** on the same node — two
separate applications of a delete by the same author:

```
"quick" → tracked_delete:Bob(id:"x")  tracked_delete:Bob(id:"y")  ← bloat
```

This is a logic error: one author cannot have two distinct pending deletes on
the same characters. The merge step must collapse these.

**Scenario B — Fragmented nodes after accept/reject**

`splitRangeForNewMark` breaks "The quick brown fox" into segments when a second
author's mark is applied. After the conflict is resolved (one author's mark
accepted/rejected), adjacent segments may end up with identical mark sets but
remain as separate nodes:

```
Before conflict resolution:
  "The "   → tracked_delete:Bob
  "quick"  → tracked_delete:Bob  tracked_delete:AI   ← conflict
  " brown" → tracked_delete:Bob

After "Reject AI":
  "The "   → tracked_delete:Bob
  "quick"  → tracked_delete:Bob                      ← now identical to neighbours
  " brown" → tracked_delete:Bob

Result without cleanup: three nodes instead of one.
```

Left uncleaned, this compounds with every conflict — documents that have had
many AI suggestions accumulate thousands of tiny text node fragments,
degrading layout performance.

### The fix — `mergeAdjacentMarks` in `appendTransaction`

Add a normalisation pass to the track-changes plugin's `appendTransaction` that
runs after any operation tagged with `TrackChangesAction.refreshChanges`.

```typescript
// packages/plugins/src/track-changes/lib/mergeAdjacentMarks.ts

/**
 * Walk every textblock in the document. For each pair of adjacent text nodes
 * whose mark sets are identical (by Mark.eq on every mark), emit a join step.
 *
 * Also removes same-author duplicate marks (same type, same authorID, different
 * id) from individual nodes — collapses them to the one with the earlier
 * createdAt.
 *
 * Must run AFTER splitRangeForNewMark and AFTER accept/reject operations.
 * Tagged with skipTrack so it does not get re-tracked by appendTransaction.
 */
export function mergeAdjacentMarks(tr: Transaction, schema: Schema): boolean {
  // Returns true if any merges were made (caller decides whether to dispatch).
}
```

**Merge rules:**

| Situation | Action |
|---|---|
| Adjacent nodes, identical mark sets (all marks eq) | Join into one node |
| Same node, `tracked_delete:Bob(id:x)` + `tracked_delete:Bob(id:y)` | Keep earlier `createdAt`, remove the other |
| Same node, `tracked_delete:Bob` + `tracked_delete:Alice` | Keep both — this is the conflict case |
| Adjacent nodes, one has Bob's mark + both-authors mark | Do not merge — sets differ |

### Where it runs

```typescript
// In trackChangesPlugin appendTransaction, after the main tracking pass:
if (docChanged || createdTr.docChanged) {
  mergeAdjacentMarks(createdTr, oldState.schema);
  createdTr.setMeta("origin", trackChangesPluginKey);
  return setAction(createdTr, TrackChangesAction.refreshChanges, true);
}
```

It also runs explicitly after `setChangeStatuses` (accept/reject), since that
is the primary source of Scenario B fragmentation.

### Performance note

`mergeAdjacentMarks` walks the full document on every tracked-change operation.
For large documents this could be slow. Optimisation: limit the walk to the
affected paragraph's range (passed as a hint from `splitRangeForNewMark`).

## Decision 2 — Conflict UX: Visual stacking + Conflict Popover ✅

### Rendering

| State | Visual |
|---|---|
| Normal tracked change | Green (insert) or Red (delete), single author tag |
| Conflict segment | **Amber background**, multiple author tags |

The `isConflict` flag drives the renderer. In `TrackChanges.ts`'s
`onEditorReady` overlay handler, when a segment has two marks and either has
`isConflict: true`, draw the amber background instead of the normal
insert/delete colour.

### Conflict popover

Clicking an amber segment opens a conflict popover:

```
┌─ Conflict in this range ─────────────────────┐
│                                               │
│  Bob    → "agile"    [Use Bob's] [Reject]     │
│  AI     → "golden"   [Use AI's]  [Reject]     │
│                                               │
│                          [Reject Both]         │
└───────────────────────────────────────────────┘
```

"Use Bob's" = accept Bob's mark, reject AI's mark on this segment, remove
`isConflict` from both.

## Conflict rules (apply-time)

When applying a new change (AI or human) that overlaps an existing mark:

| Incoming op | Existing mark | Action |
|---|---|---|
| `delete` overlaps `delete:User` — same range | Redundant | Split ranges; mark overlapping segment with both marks + `isConflict: true` |
| `delete` overlaps `delete:User` — partial overlap | Partial | Split at boundary; overlapping sub-segment gets both marks |
| `delete` overlaps `insert:User` | Collision | Split; mark segment `isConflict: true` on both marks |
| `insert` at pos inside `delete:User` range | Boundary | Insert at edge of deleted range; both coexist, no conflict |
| `insert` at same pos as `insert:User` | Adjacent | Both coexist ordered by `createdAt`; no conflict unless identical text |

## AI context format

Do **not** send the pending changes as a raw offset array. Send decorated text
that the model can reason about directly:

```
The <del author="Bob">quick </del><ins author="Bob">agile </ins>brown fox.
```

This leverages the model's ability to reason about inline diffs in text rather
than mapping abstract `{ from: 4, to: 10 }` offsets in its context window.

The full structured context per paragraph:

```typescript
interface ParagraphContext {
  nodeId: string;
  // Pseudo-XML decorated text — what the AI sees and reasons about
  decoratedText: string;
  // Clean "accept-all" text — what the AI proposes changes against
  acceptedText: string;
}
```

Example:
```typescript
{
  nodeId: "abc123",
  decoratedText: 'The <del author="Bob">quick </del><ins author="Bob">agile </ins>brown fox.',
  acceptedText:  "The agile brown fox.",
}
```

The AI tool `edit_paragraph` proposes a replacement for `acceptedText`. The
client diffs `acceptedText` → `proposedText`, maps positions via
`buildAcceptedTextMap`, and applies.

---

## `buildAcceptedTextMap` — the core primitive

Maps each character in `acceptedText` back to its exact position in the live
ProseMirror document, accounting for segments that are marked as deleted (they
exist in the doc but are absent from `acceptedText`).

```typescript
interface PosMapEntry {
  acceptedOffset: number;   // index in acceptedText
  docPos: number;            // absolute position in PM doc
  existingMarks: Mark[];     // marks on this character (tracked_insert, etc.)
}

function buildAcceptedTextMap(node: PMNode, nodeStartPos: number, schema: Schema): {
  acceptedText: string;
  map: PosMapEntry[];
  decoratedText: string;
}
```

Walk the node's inline content:
- Plain text → include in `acceptedText`, record `docPos`
- `tracked_insert` text → include in `acceptedText` (it's accepted), record `docPos`
- `tracked_delete` text → **skip** in `acceptedText`, still record `docPos` for
  boundary calculations; include in `decoratedText` as `<del author="...">`

## Implementation order

### Phase 1 — Schema (required first, everything else depends on it)
1. Add `excludes: ""` to `tracked_insert` and `tracked_delete` mark specs
   in `packages/plugins/src/track-changes/TrackChanges.ts`
2. Add `isConflict?: boolean` to `TrackedAttrs` in
   `packages/plugins/src/track-changes/types.ts`
3. Update `renderTrackedInsert` / `renderTrackedDelete` in
   `packages/core/src/renderer/` to handle `isConflict` (amber background)

### Phase 2 — Core primitives
4. `buildAcceptedTextMap(node, nodeStartPos, schema)` —
   `packages/plugins/src/track-changes/lib/acceptedTextMap.ts`
5. `diffText(a, b): DiffOp[]` — word-level LCS diff, no external dependency —
   `packages/plugins/src/track-changes/lib/diffText.ts`
6. `splitRangeForNewMark(tr, from, to, newMark)` — splits existing text nodes
   at boundaries and applies conflict detection —
   `packages/plugins/src/track-changes/lib/splitRangeForNewMark.ts`
7. `mergeAdjacentMarks(tr, schema)` — deduplicates same-author marks and
   rejoins fragmented nodes after conflict resolution —
   `packages/plugins/src/track-changes/lib/mergeAdjacentMarks.ts`
   Wire into `appendTransaction` after the main tracking pass and after
   `setChangeStatuses` (accept/reject).

### Phase 3 — AI suggestion pipeline
7. `applyDiffAsSuggestion(editor, nodeId, proposedText, authorID)` — uses 4+5+6 —
   `packages/plugins/src/track-changes/lib/applyDiffAsSuggestion.ts`
8. `buildParagraphContexts(editor)` — builds `ParagraphContext[]` for AI —
   `packages/plugins/src/track-changes/lib/buildParagraphContexts.ts`
9. Update AI endpoint: add `edit_paragraph` tool, use `buildParagraphContexts`
   output as structured context (`decoratedText` in system prompt)
10. Update `ChatPanel`: route `edit_paragraph` results through
    `applyDiffAsSuggestion`

### Phase 4 — UX
11. Wire `UniqueId` extension into the demo so every block node has a stable
    `id` attr (`packages/plugins/src/ai-toolkit/UniqueId.ts` → `App.tsx`)
12. Conflict popover in `TrackChangesPopover` — amber segments, per-author
    accept/reject buttons
