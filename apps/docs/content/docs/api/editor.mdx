title: Editor API
description: Complete reference for the Editor class — constructor, properties, methods, and lifecycle.
---

The `Editor` class is the central orchestrator of every Inscribe instance. It owns the document state, layout engine, and extension system. Consumers interact with it directly (headless) or via the `useCanvasEditor` React hook.

```ts
import { Editor, StarterKit } from '@inscribe/core';

const editor = new Editor({
  extensions: [StarterKit],
  onChange: (state) => console.log('changed', state),
});

editor.mount(containerElement);
```

---

## Constructor & Options

```ts
new Editor(options: EditorOptions)
```

Accepts an `EditorOptions` configuration object:

- **`extensions`** `(Extension[])`: Extensions that define the schema, keymap, and commands. Defaults to `[StarterKit]`.
- **`pageConfig`** `(PageConfig)`: Page dimensions and margins. Defaults to A4 with 1-inch margins.
- **`onChange`** `(EditorChangeHandler)`: Called on every state change. Optional when using React adapter (as it auto-subscribes).
- **`onFocusChange`** `((focused: boolean) => void)`: Called when the editor gains or loses focus. Use to show/hide overlays.
- **`onCursorTick`** `((isVisible: boolean) => void)`: Called every cursor blink tick (530ms) and immediately after cursor movement so the adapter can redraw the overlay.
- **`startReady`** `(boolean)`: Default `true`. When false, all render flushes are suppressed until `setReady(true)` is called. Essential for collaborative documents to prevent thousands of layout recalculations during initial Hocuspocus sync.

## Public Properties & Registries

### `schema`
```ts
get schema(): Schema
```
The merged ProseMirror schema built from all active extensions. Use this instead of importing a schema directly.

### `pageConfig`
```ts
readonly pageConfig: PageConfig
```
The active page dimensions passed to the layout coordinator and read by renderers.

### `charMap`
```ts
readonly charMap: CharacterMap
```
The spatial index mapping doc positions to `(x, y, page, height)` absolute coordinates. Used for rendering and hit-testing calculation.

### `measurer`
```ts
readonly measurer: TextMeasurer
```
The HTML5 `<canvas>` text measurer instance. Its internal width caches are reused across layout passes.

### `fontConfig` / `fontModifiers` / `markDecorators`
```ts
readonly fontConfig: FontConfig;
readonly fontModifiers: Map<string, FontModifier>;
readonly markDecorators: Map<string, MarkDecorator>;
```
Computed typography rules and mark render algorithms merged from all extensions at construction.

### `blockRegistry` / `inlineRegistry` / `toolbarItems`
```ts
readonly blockRegistry: BlockRegistry;
readonly inlineRegistry: InlineRegistry;
readonly toolbarItems: ToolbarItemSpec[];
```
Maps of block/inline rendering strategies (`measure` and `draw` loops) and the data-only toolbar specifications contributed by loaded plugins.

### `cursorManager`
```ts
readonly cursorManager: CursorManager
```
Manages the cursor blink timer. Read `cursorManager.isVisible` to know whether to draw the cursor onto your overlay canvas.

### `isFocused`
```ts
get isFocused(): boolean
```
Whether the editor's hidden native input bridge `<textarea>` currently holds focus.

### `commands`
```ts
readonly commands: Record<string, (...args: unknown[]) => void>
```
All bound extension commands (`editor.commands.toggleBold()`, `editor.commands.undo()`, etc). Closures over the Editor so they instantly read the latest state.

### `loadingState`
```ts
get loadingState(): "syncing" | "rendering" | "ready"
```
Three-phase state for active collaborative instances ensuring smooth UI progress blocks:
- `'syncing'`: Waiting for network document; no layout occurs.
- `'rendering'`: Network document received. First pages drawn; background idle callbacks actively laying out the rest.
- `'ready'`: All pages completely laid out. Non-collaborative editors start in this state immediately.

### `cursorPage`
```ts
get cursorPage(): number
```
The 1-based page number where the cursor currently resides.

## Lifecycle & Control Methods

### `setReady(ready: boolean)`
Resumes or halts the internal frame-loop rendering cycle (`requestAnimationFrame`). Used automatically by collaborative layers to halt rendering while receiving the initial bulky Y.js update.

### `mount(container: HTMLElement)`
Creates the hidden `<textarea>` input bridge and attaches keyboard/paste event listeners to the given container.

### `unmount()`
Tears down the mounted view's event listeners without destroying the internal Model state. Safe to call multiple times or remount.

### `destroy()`
Full engine teardown. Runs extension `onEditorReady` cleanup callbacks, unmounts the bridge, and destroys the layout trackers. 

### `focus()`
Programmatically focuses the hidden textarea bridge wrapper.

---

## State Access Methods

### `getState() / getSnapshot()`
Returns the current immutable ProseMirror `EditorState`. 

### `isActive(name: string, attrs?: Record<string, unknown>)`
Returns `true` if the named mark or block node type is active at the current cursor or selection range. Standard way to trigger "Active" CSS variants in toolbars.

### `getSelectionSnapshot()`
```ts
getSelectionSnapshot(): SelectionSnapshot
```
Returns a lightweight snapshot of the selection (anchor, head, from, to, activeMarks, activeMarkAttrs, blockType, blockAttrs). Provides all data required for Toolbar logic without Prosemirror imports.

### `getActiveMarks() / getActiveMarkAttrs()`
Returns strings (`["bold", "highlight"]`) or objects representing the precise marks and their attributes present exactly under the cursor gap.

### `getBlockInfo()`
Returns the parent block type (`paragraph`, `heading`) and its attributes. Traverses up out of structural wrappers (like bullet lists).

### `getMarkdown() / getMarkdownSerializer()`
Serializes the full document out to raw Markdown utilizing the extension-contributed serialization rules.

---

## Layout & View Methods

### `layout`
Forces an `ensureLayout()` recalculation and returns the full `DocumentLayout` block schema tree.

### `ensureLayout()`
Forces the internal tracking algorithm (`LayoutCoordinator`) to reflect the current Editor State.

### `getViewportRect(from: number, to: number)`
Converts a doc position range to an absolute screen-space `DOMRect` relative to the browser window. Extensively used for `@floating-ui/react` popovers.

### `getNodeViewportRect(docPos: number)`
Similar to getViewportRect, but resolves the bounding box of a Node/Block specifically, utilizing `__lookupPage` layout tracking.

### `ensurePagePopulated(pageNumber: number)`
Checks if the generic spatial CharacterMap has been fully calculated up to a target physical page; runs the calculator up if needed.

### `syncInputBridge()`
Repositions the physical hidden `<textarea>` directly under the actual visual Canvas cursor coordinate to trick native scrolling anchoring.

### `scrollCursorIntoView()`
Scrolls the active layout window/wrapper so the cursor is visually apparent.

## Subscription & Callbacks

### `subscribe(listener: () => void)`
Registers a listener called on *every* editor notification (state changes, focus changes, cursor blink). Returns an unsubscribe execution function. 

### `redraw()`
Triggers an immediate forced UI repaint notification without a State change occurring (used extensively by the awareness remote cursor plugins).

### `addOverlayRenderHandler(handler: OverlayRenderHandler)`
Registers a Canvas draw block loop on the `OverlayRenderer` tier. Returns an unregister function.
```ts
const unregister = editor.addOverlayRenderHandler((ctx, pageNum, config, charMap) => {
   // ctx.fillRect(...) 
});
```

### `setPageElementLookup(fn: (page: number) => HTMLElement | null)`
Registers the native DOM resolution function used to map the abstract layout 1-based page number to the physical browser wrapper `div` element. Used internally by layout boundary calculations.
